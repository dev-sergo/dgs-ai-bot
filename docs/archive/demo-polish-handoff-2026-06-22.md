## ЗАДАЧА 1 — case B: честный отказ на рейтинг по сотрудникам
**Проблема:** «топ продавцов / лучшие официанты / рейтинг кассиров» → модель молча отдаёт топ ТОВАРОВ
(report=products), измерения «сотрудник» в каталоге нет. Выглядит уверенно, но это не то.
**Фикс (детерминированный, как `RefineProductContribution` в internal/planner/refine.go):**
- regex на рейтинг по людям: (топ|лучш|худш|рейтинг|больше всего|меньше всего) + (продавц|официант|кассир|
  сотрудник|оператор|персонал) БЕЗ конкретного имени → честное «Рейтинг по сотрудникам пока недоступен…».
- Не задеть легальное: «чеки сотрудника Иванова» (фильтр по имени) НЕ трогать.
- Где вернуть сообщение: вариант — отдельный intent/ответ в app.Ask (как outOfScopeMessage), либо план →
  off_topic с явным текстом. Решить по месту; проще всего — пометить план и вернуть честный текст в app.
**Бенчмарк-нюанс:** кейсы «топ продавцов за прошлый месяц», «какой оператор обработал больше всего чеков»
сейчас и так FAIL (вне white-list). Их разметку пометить как ограничение (Этап 5.4), не как fixable —
pass-rate не изменится. Полный eval НЕ обязателен (правка не трогает общий промпт), но прогнать желательно.
**Тест:** юнит в planner на regex (ловит «топ продавцов», НЕ ловит «чеки сотрудника Иванова») +
pipeval-кейс «топ продавцов за месяц» → ответ содержит «недоступно», не таблицу товаров.

## ЗАДАЧА 2 — нарратив при пустом предыдущем периоде
**Проблема:** на реальном LLM нарратор для compare/contribution при value_prev=0 выдаёт «+100%» /
«доля 119%» — бессмыслица. Шаблонный нарратор (Compose) это уже обрабатывает корректно, а LLM-нарратор нет.
**Фикс:** в `internal/narrator/llm.go` (const systemPrompt) добавить правило: если предыдущий период = 0 /
данных за него нет — НЕ считать проценты/доли, написать «за предыдущий период данных не было, сравнение
неполное». Это промпт НАРРАТОРА (не планировщика) → на 551 не влияет, риска регрессии нет.
**Тест:** pipeval-host кейсы «сравни выручку с прошлой неделей», «за счёт чего вырос средний чек за неделю»
(там prev=0) → в ответе нет «+100%», есть честная оговорка. Проверяется глазами по дампу + contains.

## ЗАТЕМ — большой качественный прогон с проверкой ОТВЕТОВ
1. При желании расширить `test/eval/quality.jsonl` (сейчас 74 кейса; добавить ещё ходовых/диалогов).
2. `make pipeval-quality-host 2>&1 | tee quality-run6.txt` (~8–10 мин, реальный LLM, дамп план+ответ).
3. Прочитать quality-run6.txt и разобрать КАЖДЫЙ ответ: верны ли числа/период, сортировка, диалоги держат
   контекст, B/нарратив починены, безопасность 100%, тон годен для заказчика. Записать находки, чинить.
   Принцип: «обработалось» ≠ «ответ хороший» — проверять сами ответы, а не факт прохождения.

## Уроки (из памяти проекта)
- Правки ОБЩЕГО промпта планировщика → обязательно полный eval-host ДО коммита (узкий фикс уронил 523→507).
- Для УЗКОГО интента — детерминированная маршрутизация (regex→план), а не фаззи-правило в промпте (растекается).
- Бенчмарк плана не ловит качество ответа (order/числа/период) — нужен прогон по ОТВЕТАМ (pipeval --dump).
- «Плавающий» симптом ≠ шум: смотреть на разницу во входе (баг D — primaryMetric хватал измерение date).
