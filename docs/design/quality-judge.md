# 07 — Оценка качества ответов: LLM-судья по рубрике

Статус: **дизайн, Тир 0** (на ревью). Код не написан — это спецификация того, что будет
построено после одобрения.

## 1. Зачем

Сейчас мы умеем ловить **структурный регресс**, но не умеем измерять **качество ответа**.

Что есть:

| Инструмент | Что проверяет | Как | Покрытие |
|---|---|---|---|
| `cmd/eval` (`internal/eval`) | ПЛАН планировщика | детерминированная сверка полей (intent/report/period/filters) | ~559 кейсов |
| `cmd/pipeval` (`internal/pipeval`) | ИТОГОВЫЙ ОТВЕТ: числа, структура, утечки | детерминированно: `contains`, точное `summary`, авто-скан PII | ~25 кейсов (7 многоходовых) |
| юнит/интеграция | движок, даты, сессии, рендер | Go-тесты | ~155 функций |

Чего нет: ни один инструмент не отвечает на вопрос **«хороший ли это ответ?»** —
по существу ли он, естественный ли язык, честно ли оговорена неточность, поправляет ли
ложную посылку вопроса. `test/eval/quality.jsonl` существует, но инспектируется **глазами**
(`make pipeval-quality-host` с `PIPEVAL_DUMP=1`). Отсюда ощущение «непонятно, регрессируем
мы или улучшаемся» — оно обоснованное.

**Узкое место — не объём запросов, а оракул**: как узнать, что ответ хорош, не читая его
руками. 5000 запросов без автоматического оракула = 5000 непрочитанных ответов. Поэтому
первым строим оракул (судью), а корпус наращиваем уже под него.

## 2. Видение и критерии успеха

Мы строим **не pass/fail-ворота, а измерительный прибор** для здоровья и траектории
системы. Прибор должен отвечать на три вопроса, и только если он отвечает на все три —
он полезен:

1. **Где сейчас узко?** — в каком сценарии / по какому измерению система слабее всего.
2. **Стало лучше или хуже после изменения?** — относительно предыдущего состояния, за
   пределами шума.
3. **Куда система движется во времени?** — тренд по категориям, а не разовый снимок.

Архитектура измерения — **пирамида**, каждый слой ловит своё:

```
        ▲  человек (калибровка, выборочный аудит) — десятки, эталон истины
       ───
      ─────  LLM-судья (субъективное качество) — сотни, шумный, добавочный
     ───────
    ─────────  детерминированные eval + pipeval (план, числа, утечки) — тысячи, дёшево, точно
```

Два измерения держим **строго раздельно** (см. §3): объективная корректность считается
детерминированно, субъективное качество — судьёй. Смешение даёт ложное чувство покрытия.

Главный принцип доверия: **верь измерению ровно настолько, насколько ты измерил его
собственный шум (§7) и сверил его с человеком (§8).** Цифра без известного шумового пола и
без калибровки на человеке — не метрика, а гадание.

## 3. Принцип: judge не дублирует pipeval

Жёсткое разделение ответственности — главное архитектурное решение документа:

- **pipeval владеет ОБЪЕКТИВНОЙ корректностью.** Числа из движка, структура envelope,
  отсутствие утечек PII. Детерминированно, сверкой с фикстурами. Число судья **не** трогает.
- **judge владеет СУБЪЕКТИВНЫМ качеством.** Релевантность, полнота, естественность языка,
  честность оговорок, поправка ложных посылок. То, что нельзя выразить точным равенством.

Они не пересекаются. Если для проверки достаточно `strings.Contains` или равенства числа —
это кейс pipeval, не судьи. Судья дороже и шумнее; тратить его на то, что ловится
детерминированно, — расточительство и (хуже) внесение шума туда, где была точность.

## 4. Механика: чек-лист критериев (а не холистическая оценка)

Каждый кейс несёт **чек-лист критериев на естественном языке** — что хороший ответ обязан
сделать. Судья (отдельный вызов LLM, `temperature=0`) проверяет ответ по **каждому**
критерию: `met = true/false` + краткое обоснование.

Почему чек-лист, а не абстрактная шкала 1–5: на локальной модели reference-based оценка
(«сверь ответ с этими требованиями») устойчивее и прозрачнее холистической — у судьи нет
свободы выдумывать шкалу, каждое решение привязано к явному требованию и аудируемо. Бинарные
решения к тому же стабильнее между прогонами (см. §7). Холистический `overall` (1–5) тоже
собираем, но как **дополнительный** сигнал; зачёт кейса считается детерминированно из
чек-листа, а не из доверия слову судьи.

Поток одного кейса:

```
кейс → проиграть history в свежей сессии → app.Ask(query) → текст ответа
     → судья(вопрос + контекст + ответ + чек-лист) → вердикт по каждому пункту
     → pass/fail считаем сами из вердикта
```

Генерация ответа идёт через настоящий `app.Ask(ctx, tenant, session, text)` на
**фикстурах** с фиксированным «сейчас» (`2026-06-19 10:00 UTC`, как в pipeval и
интеграционных тестах) — числа предсказуемы, прогон детерминирован со стороны данных.

## 5. Формат кейса (`test/eval/judge/*.jsonl`)

Один JSON-объект на строку (как в pipeval). Поля:

| Поле | Тип | Смысл |
|---|---|---|
| `id` | string | стабильный идентификатор кейса (для A/B-diff между прогонами) |
| `scenario` | string | `structure` \| `forecast` \| `followup` — группировка отчёта |
| `history` | []string | предыдущие реплики пользователя (многоходовой контекст); опц. |
| `query` | string | сам вопрос |
| `criteria` | []string | что хороший ответ ДОЛЖЕН сделать (проверяется КАЖДЫЙ) |
| `must_not` | []string | чего быть НЕ должно (ложные утверждения, выдуманные числа); опц. |
| `severity` | string | `critical` \| `major` \| `minor` — вес кейса в метрике (по умолч. `major`; см. §8) |

### Примеры (3 выбранных сценария)

**Структура продаж** — сравнение, где выручка на самом деле упала (3304→2471):
```json
{
  "id": "structure-compare-month",
  "scenario": "structure",
  "severity": "critical",
  "query": "сравни выручку за этот месяц",
  "criteria": [
    "Ответ называет обе сравниваемые величины (текущий и прошлый период) или их разницу",
    "Ответ корректно указывает направление изменения — снижение, а не рост",
    "Язык естественный и понятный владельцу бизнеса, без сухого канцелярита"
  ],
  "must_not": ["Утверждение, что выручка выросла"]
}
```

**Структура продаж** — ложная посылка в вопросе (выручка упала, а спрашивают «почему выросла»):
```json
{
  "id": "structure-false-premise",
  "scenario": "structure",
  "severity": "critical",
  "query": "почему выросла выручка за этот месяц",
  "criteria": [
    "Ответ поправляет посылку вопроса: выручка не выросла, а снизилась",
    "Ответ опирается на конкретные числа, а не общие слова"
  ],
  "must_not": ["Ответ соглашается с тем, что выручка выросла"]
}
```

**Прогноз** — run-rate проекция:
```json
{
  "id": "forecast-month-end",
  "scenario": "forecast",
  "severity": "major",
  "query": "сколько будет выручки к концу месяца",
  "criteria": [
    "Ответ даёт конкретное прогнозное число выручки к концу периода",
    "Ответ помечен как прогноз/оценка, а не подан как свершившийся факт",
    "Если данных мало (неполная неделя), ответ оговаривает погрешность прогноза"
  ]
}
```

**Многоходовой** — наследование контекста:
```json
{
  "id": "followup-revenue-period",
  "scenario": "followup",
  "severity": "major",
  "history": ["выручка за неделю"],
  "query": "а за прошлый месяц?",
  "criteria": [
    "Ответ понял, что речь по-прежнему о выручке (унаследовал предмет из контекста)",
    "Ответ относится к прошлому месяцу, а не к неделе",
    "Ответ не переспрашивает то, что уже ясно из контекста диалога"
  ]
}
```

Все кейсы привязаны к данным, которые реально есть в фикстурах (см. `test/eval/pipeline.jsonl`
как референс рабочих запросов).

Кроме кейс-специфичных критериев к **каждому** ответу применяется небольшой набор
**универсальных** критериев (нет выдуманных чисел, ответ отвечает на заданный вопрос, связный
русский без канцелярита, нет утечек) — базовая сетка против забытых измерений (§8).

## 6. Формат вердикта и логика зачёта

Судья возвращает строгий JSON:
```json
{
  "criteria":   [{"met": true,  "reason": "…"}],
  "violations": [{"present": false, "reason": "…"}],
  "overall": 4,
  "summary": "одно предложение — главный вывод"
}
```
`criteria`/`violations` — в том же порядке и количестве, что в кейсе (сверяем по индексу,
устойчиво к кривому ответу модели).

**Зачёт кейса (детерминированно, не по слову судьи):**
```
pass = ВСЕ criteria.met == true
     И НИ ОДНО violations.present == true
     И overall >= MIN_OVERALL   (порог, по умолчанию 3)
```

## 7. Однородность результатов: откуда шум и как его ограничиваем

Тестирование **будет каждый раз немного разным** — это надо не игнорировать, а измерить и
ограничить. Источников разброса два, и они складываются:

- **(а) Генерирующая LLM нестабильна даже при `temp=0`.** `llama.cpp` на риге не
  битово-воспроизводим: батчинг, KV-кэш, конкурентные запросы → иногда другой токен →
  другой план/текст. Это зафиксированный шум eval-host (~9 кейсов). Он *выше* судьи —
  варьируется сам ответ.
- **(б) Судья — вторая стохастическая прослойка.** На пограничном критерии судья может
  «перекинуться» даже на идентичном ответе. Два шумных слоя друг на друге.

Меры (по убыванию эффекта):

1. **Зафиксировать всё фиксируемое.** `temp=0` — уже есть. **Seed мы сейчас не шлём** (в
   `internal/llm.Client` поля seed нет) — это рычаг: добавить `seed` в запрос. Плюс одиночный
   поток без конкурентных запросов, прибитые build/quant/context-size модели, фиксированные
   фикстуры и «сейчас».
2. **Бинарные + чётко сформулированные критерии.** Главный источник дрожания судьи — *размытые*
   критерии. Чёткость формулировки = воспроизводимость; это работа над корпусом, не над кодом.
3. **Голосование** (`JUDGE_VOTES=N`, majority): денойз пограничных решений.
4. **Измерить шумовой пол ДО того, как мерить изменения.** Разовая калибровка test-retest:
   прогнать один корпус K раз *без изменений*, посчитать долю кейсов с нестабильным вердиктом.
   Это даёт доверительный интервал рига. Реальное изменение обязано сдвинуть метрику *за*
   пределы пола — иначе это не сигнал. Без этого «507 vs 523» — гадание.
5. **Разделять flaky и failing.** Кейс, скачущий между прогонами, — *проблема измерения*;
   стабильно падающий — *проблема системы*. Отчёт показывает три корзины: стабильно прошло /
   стабильно упало / нестабильно. Смешение этих корзин в одном pass-count и убивает однородность.
6. **Относительное вместо абсолютного — парный A/B.** Один корпус через версию A и версию B в
   одних условиях; общий шум сокращается. Это главный измеритель «лучше/хуже», а не абсолютный
   счёт. (Тир после Тир 0.)

## 8. Угрозы валидности и как их закрываем

Дизайн §1–§7 — необходимый, но **недостаточный** каркас. Ниже — слабые места самого подхода и
встроенные ответы на них.

### 8.1. Корпус тестирует только то, что мы придумали

Курированный корпус отражает *нашу* модель использования, не реальную. Самые опасные баги —
там, где спросили то, что нам в голову не пришло; чек-лист не может провалить критерий,
которого в нём нет.

Закрываем:
- **Майнинг реальных логов.** В конфиге уже есть `QUERY_LOG_PATH` (JSONL вопросов/ответов).
  Корпус должен *сходиться к реальному распределению* запросов пилота, а не жить в голове
  автора. Это рычаг №1 по покрытию.
- **Универсальные критерии** на каждый кейс (§5) — базовая сетка против забытых измерений.
- **Открытый критик-проход.** Отдельный промпт судьи: «что не так с этим ответом, чего нет в
  чек-листе?». Ловит *unknown-unknowns* — ровно то, против чего чек-лист бессилен. Его находки
  становятся кандидатами в новые критерии/кейсы.
- **Синтетические краевые фикстуры.** Фикстуры заморожены и узки (пустые периоды, гигантские
  числа, ничьи, перевес возвратов не встречаются). Добавляем наборы под краевые формы данных.
  Тут есть напряжение с §7 (больше форм → шире поведение → больше шума) — поэтому краевые
  наборы держим отдельным, помеченным срезом.

### 8.2. Судья — та же система в другой шляпе

LLM, оценивающая LLM (тем более родственной семьи), имеет **коррелированные** слепые пятна:
оба могут разделять одно заблуждение и оба его не заметить. Судья не прыгнет выше своей
компетентности — это фундаментальный потолок.

Закрываем:
- **Человек в калибровочной петле.** Судья достоин доверия ровно настолько, насколько
  коррелирует с человеком на выборке. Периодически вручную сверяем 10–20 вердиктов и меряем
  согласие (agreement). Падает согласие — судье нельзя верить, чиним рубрику/модель.
- **Gold-set с известным правильным вердиктом** — ловит **дрейф самого судьи** при смене
  модели рига (тот же ответ → тот же вердикт?).
- **Судья ≠ оцениваемая модель** (`JUDGE_MODEL`) — по возможности сильнее/другая.
- **Детерминированно — всё, что детерминируемо.** Не спрашивать судью то, что считает pipeval.

### 8.3. Goodhart: переобучение под корпус

Как только корпус стал целью, правки промпта тюнятся *под него*, и корпус перестаёт отражать
реальность.

Закрываем:
- **Ротация held-out набора** из свежих логов — часть кейсов не показываем себе при тюнинге.
- **Регулярное обновление корпуса** из новых логов.
- **Не тюнить напрямую против эталонного eval’а**, который держим как ground truth.

### 8.4. Диагностичность вместо одной цифры

Скаляр «прошло 18/20» не говорит ни о здоровье, ни о траектории.

- **Разбивка, не скаляр:** под-оценки по сценарию, по типу провала, по критерию.
- **Таксономия провалов:** выдуманное число / не тот отчёт / проигнорирован контекст / прогноз
  без оговорки / отказ на валидный вопрос / утечка / корявый язык. Тренд по категориям во
  времени = ответ на «как система развивается».
- **Тяжесть, а не только pass/fail:** `severity` кейса (§5). Неверное число (`critical`) и
  суховатый тон (`minor`) не равноценны; метрика взвешивается по тяжести, иначе тонет в косметике.
- **Храповик регрессий:** однажды прошедший кейс при будущем фейле = алярм.

## 9. Объём Тир 0

- Стартовый корпус: **~15–25 курированных кейсов**, поровну по трём сценариям
  (`structure` / `forecast` / `followup`). Качество корпуса важнее количества: 20 точных кейсов
  с хорошими критериями дают больше сигнала, чем 2000 случайных.
- Чек-лист (§4) + универсальные критерии (§5) + `severity` (§8.4) — в Тир 0.
- Голосование по умолчанию выключено (`JUDGE_VOTES=1`) — включаем, когда увидим разброс.
- **Не** в Тир 0 (следующие тиры): парный A/B, майнинг логов, открытый критик-проход,
  gold-set дрейфа, синтетические краевые фикстуры. Дизайн под них заложен здесь, чтобы строить
  без переделок.

## 10. Артефакты, которые будут построены (Тир 0)

1. `internal/judge/judge.go` — типы, поток `Run`, агрегация голосов, `LLMScorer`, разбор
   вердикта, `Summarize` (с разбивкой по сценарию и взвешиванием по `severity`). Судья за
   интерфейсом `Scorer` — юнит-тестируется фейком без рига.
2. `cmd/judge/main.go` — команда (обвязка по образцу `cmd/pipeval/main.go`).
3. `test/eval/judge/{structure,forecast,followups}.jsonl` — стартовый корпус.
4. `internal/judge/judge_test.go` — детерминированные юнит-тесты на разбор/агрегацию/зачёт/
   взвешивание (гоняются в обычном `make test`, рига не требуют).
5. `Makefile` — цель `judge-host` по образцу `pipeval-host`.

Прод-код (`app`/`planner`/`engine`/`narrator`) **не трогается** — судья изолирован. Исключение
для §7.1 (seed) — добавление поля `seed` в `internal/llm` — отдельным, явно согласованным шагом.

## 11. Как будет запускаться (host-only, нужен риг)

Прогон против живой LLM запускает пользователь (память `feedback-user-runs-rig`), вывод — в
лог-файл (`feedback-save-run-output`):

```
make judge-host 2>&1 | tee judge-$(date +%Y%m%d-%H%M).log
# переменные: JUDGE_CASES, JUDGE_MODEL, JUDGE_VOTES, JUDGE_MIN_OVERALL, JUDGE_DUMP=1
```

## 12. Дорожная карта (за пределами Тир 0)

1. **Парный A/B-режим** (§7.6) — главный измеритель «лучше/хуже».
2. **Калибровка шумового пола** (§7.4) — разовый test-retest, чтобы знать доверительный интервал.
3. **Майнинг логов** (`QUERY_LOG_PATH`, §8.1) — корпус сходится к реальному распределению.
