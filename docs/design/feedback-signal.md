# 08 — Сигнал обратной связи (👍/👎) из UI и Telegram

Статус: **дизайн** (на ревью). Код не написан.

## 1. Зачем

Замыкаем петлю обратной связи из [docs/07](07-quality-judge.md) (§8.1, ступень 6 «маховик
данных»). Сейчас захват есть — `querylog` пишет «вопрос → план → ответ»
([querylog.go](../internal/querylog/querylog.go)), — но **сигнала качества от пользователя
нет** (греп по feedback/rating пуст). Без него логи — мёртвый архив. Цель: собрать простой
бинарный сигнал 👍/👎 **из обоих каналов** (UI и Telegram) в один датасет.

Принцип: сигнал должен быть максимально дешёвым для пользователя (один тап) и одинаковым в
обоих транспортах. Никакой аналитической ценности в «богатом» фидбэке на старте нет — нужен
поток, а не глубина. Свободный комментарий — осознанно ВНЕ объёма (см. §9).

## 2. Ключевое решение: ID ответа

Оценку надо привязать к конкретному ответу. Сейчас ни `Answer`, ни `querylog.Record` не несут
идентификатора. Вводим **короткий ID ответа**:

- генерируется в `app.Ask` на каждый ответ;
- кладётся в строку `querylog` (датасет вопросов) И возвращается клиенту в `Answer`;
- фидбэк пишется ОТДЕЛЬНОЙ строкой `{id, rating}` в свой лог; связь «оценка ↔ вопрос/ответ»
  восстанавливается офлайн джойном по `id` (см. §8).

Почему отдельный лог, а не правка строки querylog: JSONL — append-only, дозапись на месте
невозможна и не нужна. Два потока, джойн по ключу — стандарт и проще.

Требования к ID: короткий (callback_data в Telegram ≤ 64 байт), уникальный в пределах
датасета. Генератор **инжектируемый** (поле `App.NewID`, по умолчанию — случайный/монотонный),
чтобы детерминированные тесты (pipeval/интеграция) могли его зафиксировать или игнорировать.

## 3. Модель данных

### 3.1. Дополнение querylog
В [querylog.Record](../internal/querylog/querylog.go#L23) добавляется поле:
```go
ID string `json:"id"` // короткий идентификатор ответа (ключ для джойна с feedback)
```

### 3.2. Новый пакет `internal/feedback`
Копия структуры `querylog` (потокобезопасный JSONL-Writer, nil = выключено):
```go
type Record struct {
    TS      string `json:"ts"`              // RFC3339
    ID      string `json:"id"`              // ID ответа, к которому относится оценка
    Tenant  string `json:"tenant"`
    Session string `json:"session"`
    Rating  string `json:"rating"`          // "up" | "down"
    Source  string `json:"source"`          // "ui" | "telegram"
}
```
Одна строка на оценку. Дубль (пользователь тапнул дважды) пишем как новую строку —
дедуп/«последнее слово» решаем офлайн, транспорт не усложняем.

## 4. Поток (оба канала)

```
app.Ask → Answer{ID, …}             (ID заодно ушёл в querylog-строку)
   ├─ UI:        ответ показан + кнопки 👍/👎 → POST /feedback {id, rating} → feedback.jsonl
   └─ Telegram:  ответ + инлайн-кнопки (callback_data="fb:up:<id>") → CallbackQuery → feedback.jsonl
```

Запись фидбэка идёт через единый метод `app.RecordFeedback(id, tenant, session, rating, source)`
— оба транспорта используют его, логика и тест в одном месте.

### 4.1. Где показываем кнопки

Правило: **на всех ответах, КРОМЕ инфраструктурных ошибок** (отчёт с данными, «данных нет»,
совет, переспрос, help, smalltalk, отказ). Ошибку (`outcome=error`) не оцениваем — это сбой
инфраструктуры, а не качество ответа.

Важно про smalltalk/отказы: 👎 там означает **не «плохое приветствие», а мисроутинг** — бот
ответил «привет»/отказом на реальный вопрос. Это ценный сигнал «ты меня не так понял», поэтому
оцениваем и их. «Мусор» (кто-то минуснул честный «привет») режется офлайн по `intent`, который
уже пишется в `querylog` (`jq 'select(.intent=="smalltalk")'`).

Это правило отображения — один предикат на стороне UI/бота, **обратимое**: если smalltalk-оценки
окажутся шумом, область сужается одной строкой без переделок.

## 5. Контракт HTTP

Новый роут под тем же auth-гейтом, что `/ask`:
```
POST /feedback
body: {"id": "<answer id>", "rating": "up" | "down"}
→ 200 {"status":"ok"}   |  400 (нет id / неверный rating)
```
tenant/session — из заголовков `X-Tenant-ID` / `X-Session-ID` (как в `/ask`).

- [bot.go](../internal/transport/telegram/bot.go) `ask()`: к сообщению-ответу прикрепляется
  `InlineKeyboardMarkup` с двумя кнопками; `callback_data = "fb:up:<id>"` / `"fb:down:<id>"`.
- `Run()`: сейчас цикл берёт только `upd.Message` и игнорирует остальное. Добавляется ветка
  `upd.CallbackQuery`: разбор `fb:<rating>:<id>` → `app.RecordFeedback(...)` → `AnswerCallbackQuery`
  (всплывающий toast «Спасибо!») → опционально снять/обновить клавиатуру, чтобы не голосовали дважды.

## 8. Конфиг и чтение метрик

- Включение: `FEEDBACK_LOG_PATH` (по аналогии с `QUERY_LOG_PATH`); пусто → выключено.
- `bootstrap` открывает лог, кладёт в `app.FeedbackLog`, возвращает в общий cleanup.
- Чтение (офлайн, без кода):
```bash
# доля 👍 за всё время
jq -s 'group_by(.rating)|map({(.[0].rating): length})|add' feedback.jsonl

# вопросы, получившие 👎 (джойн с датасетом по id)
jq -s '[.[]|select(.rating=="down")|.id]' feedback.jsonl > down.json
jq --slurpfile d down.json 'select(.id as $i|$d[0]|index($i))' queries.jsonl
```
Это даёт первый реальный сигнал «где система проседает» — топливо для корпуса судьи (docs/07).

## 9. Оговорки и зависимости (важно)

1. **👎 — это сигнал-фильтр, а не метрика качества.** Большинство пользователей не тапнут
   ничего; реагируют выборочно и эмоционально → выборка смещённая. **Нельзя считать
   «% положительных» и выдавать за здоровье системы** (прямой путь к самообману, docs/07 §8.3).
   Ценность 👎 в другом — это *очередь кейсов, которые читаем глазами* и превращаем в кейсы
   судьи (docs/07). Агрегат — вторичен, выборка для разбора — первична.
2. **Фидбэк бесполезен без `QUERY_LOG`.** `feedback.jsonl` несёт только `{id, rating}` — без
   датасета вопросов джойнить не с чем. На старте, если включён `FEEDBACK_LOG_PATH` без
   `QUERY_LOG_PATH`, пишем предупреждение в лог (оценки копятся «в вакууме»).
3. **В UI сейчас не шлётся `X-Session-ID`** ([api.ts](../web/src/api.ts) шлёт только tenant) →
   серверная сессия у всех веб-пользователей общая (`default:mock_single`). Значит у
   `source="ui"` поле `session` низкого разрешения. Для MVP не блокер — джойн идёт по уникальному
   `id`; при желании UI позже начнёт слать стабильный `X-Session-ID`.

## 10. Вне объёма (осознанно)

- Свободный комментарий при 👎 (особенно ввод текста в Telegram) — позже, если поток 👎 пойдёт.
- Пер-критерий / звёзды / NPS — нет, бинарный сигнал достаточен для старта.
- Дашборд метрик — пока `jq` по логам; агрегатор позже.
- Дедуп/«последнее слово» по повторным тапам — офлайн, не в транспорте.
- Привязка к онлайн-A/B — после того как наберётся объём.

Telegram `CallbackQuery` (новый тип апдейта; паттерн стандартный). Прод-математика и sandwich
не затрагиваются.
