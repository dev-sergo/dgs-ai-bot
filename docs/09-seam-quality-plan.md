## Зачем эта итерация

Симптом, с которого начали: **тесты зелёные, а при ручном тестировании бота всплывает много багов**
(в первую очередь фильтры). Это не паранойя и не «мало тестов вообще» — это структурная вещь.

**Диагноз.** У проекта здоровое ядро (sandwich: числа из движка/API, LLM — план+нарратив) и хорошо
покрытый детерминированный слой (engine, dates, validate, golden-тесты dooglys). Но **пирамида тестов
перевёрнута**: интеграционные тесты в `test/integration/` гоняют полный путь через `StubPlanner` —
то есть план захардкожен, **живой LLM не участвует никогда**. А баги живут ровно на **шве** — границе,
где живой текст пользователя превращается в детерминированный план:

```
текст → LLM-планировщик → refine (Go-эвристики) → validate (whitelist) → resolver (имя→UUID) → Query
```

Всё до шва — хаос естественного языка; всё после — надёжная математика (протестирована). Качество
продукта в глазах заказчика определяется почти целиком швом: математика не ошибается, а шов — да,
и юзер сразу видит «спросил про карту — показало всё» / «фильтр Выкса потерялся».

## Главный принцип итерации

**Ноль новых отчётов / методов / каналов.** Каждая фича в ширину добавляет ещё один непротестированный
путь через тот же шов. Эта итерация — только глубина: планировщик стабильнее, refine не врёт, состояние
не теряется, имена резолвятся, дырявые планы не проходят, промахи видны в логах.

## Что вскрыл прогон 100 запросов (`batch-20260625-1556.log`)

Ручной прогон 100+ запросов через ЖИВОЙ LLM (фактически — ручная версия Задачи 7). Итог: **~78-80%
верной маршрутизации**, но провалы СОСРЕДОТОЧЕНЫ:
- **Ядро железобетонно (~95%+):** выручка/период/средний чек, сравнения, top_n товаров, каналы оплаты, off-topic/отказы.
- **Прогноз — главный провал (~30%):** `forecastRe` слишком узкий, ~18 формулировок ушли в compare/plain
  вместо forecast. Не ловит: «план/цель N — выйдем/дойдём/реально/успеем» (32-42, 92-93); порядок слов
  (33 «цель 1 млн — дойдём?» — якорь до глагола); синонимы («предсказание» 25, «прогноз на конец» 22);
  «к концу июня» (26, regex знает месяц/неделя/период, не названия месяцев); «при текущем темпе / если темп
  сохранится» (19, 24, 28).
- **goalRe не извлекает цель даже когда forecast сработал** (30 «до плана **в 2 миллиона**», 31 «в 1.5 млн»):
  ответ generic, без упоминания цели. Причина: regex ждёт «план » + пробел, ломается на «план**а**»; ветка
  «дойти до» не ловит «дойду». Плечо `ForecastEnvelope(goal)` готово — не работает ТОЛЬКО извлечение.
- **Parse-fail под видом smalltalk (узел #6!):** 69 «помоги мне», 71 «спасибо», 72 «пока», 81 «дойду ли я до
  плана», 82 «дойдём ли» → ответ «Не смог разобрать запрос» (строка из ветки parse-fail `llm.go:53`).
  Это НЕ баг render — это битый JSON от модели, маскирующийся под smalltalk. В `outcome`-логе невидим →
  ровно цель Задачи 1.
- **Бэклог (НЕ в этой итерации):** advice/report протекает (62 «что сделать чтобы увеличить продажи»,
  63 «дай рекомендации» → contribution-отчёт вместо advice). Advisor заморожен — паркуем.

**Вывод:** прогон НЕ опроверг план — усилил его. Самый дешёвый и видимый на демо первый ход —
не абстрактный hardening, а **прогноз (Задача F)**, т.к. это фича, которую увидит заказчик.
