## Блок Т — тестовый контур (расширение покрытия; параллельно Блоку 0 и после)

Цель: максимальное покрытие автоматикой, чтобы ничего не гонять руками.
Дешёвый eval (3–4 мин) делает это реалистичным.

- [ ] **Т1. Матрица покрытия планировщика.** Разложить prompts.jsonl по осям
  «отчёт × метод × период × фильтры» и добить дыры кейсами. Известные слабые места:
  categories (модель роняет фильтры — видно в фейлах прогона 0fee283), new-reports (всего 16),
  cash-формы, перефразировки/кривые формулировки.
- [ ] **Т2. Диалоги и контекст.** Расширить `pipeline-followups.jsonl` (сейчас 7 кейсов)
  до полноценного диалогового корпуса: carry-over периода/фильтров между ходами,
  цепочка confirm (полоса 0.5–0.7 → «да»), clarify → ответ, advice-resume (AwaitingPeriod),
  смена темы сбрасывает контекст, длинный диалог (история капится MaxMessages —
  проверить, что промпт не пухнет), изоляция сессий и тенантов (нет утечки истории).
- [ ] **Т3. Тесты на живых данных (не только фикстуры).** Режим `pipeval-live`:
  прогон через реальный Report-API (`DGS_CLIENT=api`) с **инвариантными** ассертами вместо
  захардкоженных чисел (числа меняются каждый день): непустые rows, mentions_number,
  структура envelope, согласованность (сумма строк = summary; производная движка сходится
  с totals Report-API). Малый набор (10–20 кейсов), запуск перед демо/релизом; не долбить
  прод-API конкурентно.
- [ ] **Т4. Генератор кейсов — не писать руками.** Перенести подход `scripts/gen_cases.py`
  (детерминированные синтетические кейсы, seed) на Go-eval корпуса: шаблоны запросов ×
  сущности × периоды × шум формулировок → авто-прирост prompts.jsonl. Плюс майнинг
  из querylog пилота: 👎-записи → черновики кейсов (см. Блок 4).
- [ ] **Т5. Стресс и «плохой ввод».** Что уже есть: HTTP body cap 1 MiB + тест на 413;
  история сессии капится. Чего нет (добавить тесты):
  - гигантский запрос (до cap 0.4) → вежливый ответ, не 500/не LLM-ошибка;
  - запрос-мусор / вставка «простыни» данных / prompt-injection-простыня → fail-closed
    (smalltalk/clarify), проверка refuse-префильтра на больших входах;
  - переполнение контекста LLM: длинный запрос + длинная история → деградация graceful;
  - огромный результат: top-N без лимита, выгрузка xlsx на тысячах строк — рендер и память;
  - флуд одним пользователем (rate-limit в Telegram-боте) — поведение и текст ответа.
- [ ] **Т6. Тулинг под быстрый eval.** (а) model-tag автоматически в JSON-сводку прогона
  (cmd/eval) — чтобы смена модели не читалась как регресс кода; (б) per-category разбивка
  фейлов в summary (где именно просело); (в) обновить [BENCHMARKS.md](BENCHMARKS.md):
  тайминги 3–4 мин, политика «полный прогон на каждую правку», новые корпуса из Т1–Т3.
