# Глоссарий — понятия проекта (LLM/AI + архитектура)

> Зачем: чтобы не путаться в терминах и говорить с тех-стороной на одном языке.
> Особенно важно одно различие, на котором мы уже спотыкались:
> **MCP = труба к данным** (интерфейс), **ядро/движок = мозг** (планирует и считает). Это РАЗНОЕ.

---

## Путь запроса: этапы (как зовутся и почему)

Поток одной строкой:
**текст → планировщик → parse → refine → validate → resolver → engine → нарратор** (вокруг — session).

| Этап | Что делает | Почему так называется | Где в коде |
|---|---|---|---|
| **Канал / входной адаптер** | принимает сырой текст (веб-чат, Telegram) | «канал» доставки сообщения | адаптеры канала |
| **Планировщик (planner)** | LLM превращает текст в *структурный план* (что достать/посчитать), **не ответ** | строит *план* действий, а не финальный ответ | `internal/planner/llm.go` |
| **Парсинг (parse)** | разбор JSON-ответа модели в Go-структуру; *parse-fail* = битый JSON | parse = «разобрать» текст в структуру | `internal/planner/llm.go` (`parsePlan`) |
| **Refine (доводка)** | **Go-гарды** чистят/правят план | refine = «довести, уточнить» | `internal/planner/refine.go` |
| **Validate (валидатор)** | проверяет план по **whitelist** | от validation = «проверка допустимости» | `internal/plan/validate.go`, `plan.go` |
| **Resolver (резолвер)** | имя человека → ID объекта («Выкса» → UUID) | resolve = «разрешить/сопоставить» имя реальному объекту | `internal/resolver/resolver.go` |
| **Engine (движок)** | детерминированная математика, выдаёт числа | «движок» расчётов — ядро системы | engine-слой |
| **Нарратор (narrator)** | LLM оборачивает числа в человеческие слова | narrate = «рассказать» | планировщик/нарратив |
| **Session (сессия)** | память диалога между ходами | «сессия» = непрерывный диалог | `internal/session/` |

Оркестрация этапов: `internal/app/app.go` (`resolveFilters`, история в планировщик и т.д.).

## A. Архитектурные понятия

- **Go-гарды (guards)** — маленькие детерминированные функции на **Go** в `refine.go`, которые «сторожат» план от ошибок модели: ловят конкретный случай и правят/блокируют его (как *guard clause* в программировании — проверка-страж в начале функции). «Go» — потому что обычный код, не LLM; «гард» — потому что охраняет от кривого плана. Сейчас их 8 (каналы оплаты, прогноз и т.д.). В целевой схеме большинство **уходит** — их роль берут хорошие описания инструментов.
- **Sandwich (бутерброд)** — LLM сверху (понять кривой вопрос) + LLM снизу (объяснить), а **в середине твёрдое детерминированное ядро** (числа). Мягкие слои снаружи, твёрдый внутри → гибкость + доверие. Наш ключевой принцип.
- **Детерминированный** — одинаковый вход всегда даёт одинаковый выход; ноль случайности и выдумок. Про код, не про модель.
- **Whitelist (белый список)** — список разрешённого (отчёты/метрики). Всё, чего нет в списке, отвергается. Противоположность blacklist.
- **Plan (план)** — структурное представление намерения (отчёт, метрики, фильтры, период). То, что выдаёт планировщик.
- **Schema (схема)** — формальное описание структуры (какие поля и типы) плана/аргументов; по ней идёт валидация.

## B. LLM / агентные понятия

- **LLM** — large language model, большая языковая модель: понимает и генерирует текст.
- **Галлюцинация** — когда модель выдаёт правдоподобное, но **неверное** (выдуманное число/факт). Главная причина, почему числа считаем кодом.
- **Prompt (промпт)** — инструкция/запрос модели. **System prompt** — постоянная «конституция» поведения; **user prompt** — конкретное сообщение пользователя.
- **Tool / function calling (вызов инструментов/функций)** — модель не считает сама, а **вызывает заранее описанные функции**; система их исполняет и возвращает результат модели. Ключ к гибкости целевой архитектуры.
- **Оркестратор / agent-loop** — цикл, где LLM может вызвать несколько инструментов подряд, смотреть на промежуточные результаты и решать следующий шаг, пока не соберёт ответ. «Агент» — потому что действует сам, многошагово.
- **Tool registry (реестр инструментов)** — каталог доступных инструментов с описаниями, который «видит» модель.
- **MCP (Model Context Protocol)** — стандарт, как внешний сервис **выставляет данные/инструменты** для модели. = труба к данным Dooglys (её собиралась писать тех-сторона). **MCP ≠ наше ядро.**
- **Context (контекст)** — всё, что подаётся модели за один вызов (запрос + история + данные). Имеет лимит размера.

---

## C. Тестовые понятия (зачем мы их гоняем)

- **Eval / eval-harness** — прогон **корпуса** тестовых запросов через систему, чтобы измерить качество («harness» = тестовая обвязка).
- **Corpus (корпус)** — набор примеров запросов с ожидаемым результатом.
- **Fixtures (фикстуры)** — заранее заготовленные данные/ответы для тестов вместо живого API.
- **Golden-тесты** — сверяют вывод с зафиксированным **эталоном** (golden = эталонный): если вывод изменился — тест краснеет.

---

## D. Два уровня архитектуры (как сейчас и куда движемся)

**Сейчас (линейный конвейер):** планировщик выдаёт один жёсткий план, Go-гарды его латают, whitelist проверяет. Надёжно, но новая фича = новый код → медленно.

**Целевое (оркестрация инструментов, «как Restik», но с нашим детерминизмом):** LLM-оркестратор вызывает универсальные инструменты (`query_metrics`, `resolve_filter`, действия), числа приходят из реального исполнения инструмента, write-действия проходят **шлюз подтверждения**. Гибкость берётся из универсальных инструментов + каталога-данных, доверие — из детерминированного исполнения и строгого ядра для денег-метрик.
