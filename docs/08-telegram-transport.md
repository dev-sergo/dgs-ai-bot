# 08 — Telegram-транспорт

Статус: **реализовано** (адаптер — коммит 741ad06; вынос в отдельный сервис — 2026-06-25).
Тонкий адаптер поверх существующего ядра; план по этапам в §5. Транспорт живёт **отдельным
процессом** `cmd/bot` (общая сборка `*app.App` — `internal/bootstrap`), независимо от HTTP-сервера.

## 1. Зачем и принцип

Telegram — родной канал аудитории (управляющие/владельцы RU-общепита) и **поверхность сравнения**:
бот заказчика («Петя Финансист») живёт в Telegram, и демо в том же канале — честное сравнение по
сути, а не по формату. Slack из [00-pilot-vision.md](00-pilot-vision.md) (месяц 1) заменяется на
Telegram как первичную обёртку.

**Бот — тонкий адаптер, не проект.** Вся логика (NLU, математика, advisor, Excel) уже за чистым
контрактом `app.Ask(ctx, tenantID, sessionID, text) → Answer{Text, Envelope}`
([internal/app/app.go](../internal/app/app.go)). Бот только: принять апдейт → вызвать `Ask` →
отрендерить `Answer`. Бизнес-логики в боте — ноль.

## 2. Развязка с ядром (контракт)

Зависимость односторонняя через один узкий стабильный шов — тот же, что уже обкатан HTTP-транспортом:

- Изменения **за контрактом** (планировщик, математика, новый отчёт, advisor) → бот получает их
  **бесплатно**, без правок. Завели прогноз в ядре — бот начинает его транслировать сам.
- Изменения **самого контракта** (сигнатура `Ask` / форма `Answer`/`Envelope`) → правят оба
  транспорта (и бот, и HTTP). Редкость: контракт общий и стабильный.

Масштабирование по осям: транспортные фичи (команды, голос, формат) растут в пакете `telegram/`;
контентные (прогноз, мультиточка, отчёты) — в ядре, бот их транслирует прозрачно.

## 3. Базовые решения

| Решение | Выбор | Почему |
|---|---|---|
| Сборка | отдельный бинарь `cmd/bot` над общей сборкой `internal/bootstrap.App`; HTTP — `cmd/server` | бот развивается, рестартует и деплоится независимо от API; ноль дублирования проводки `*app.App` |
| Библиотека | `github.com/go-telegram-bot-api/telegram-bot-api/v5` | самая обкатанная, простой long-polling |
| Режим отправки | plain-текст (без Markdown) в v1 | обходим экранирование MarkdownV2 |
| Доставка апдейтов | long-polling, не webhook | публичный URL не нужен |
| Tenant/точка | команда `/tt КОД` → фильтр `sale_point` в сессии | как у Пети; см. §6 про границу |
| Allowlist | guard пишем сразу, пустой `TELEGRAM_ALLOWLIST` = открыт всем | копия паттерна `AuthToken` ([transport/http/server.go](../internal/transport/http/server.go)); ограничение включается env-переменной без кода |

## 4. Подача отчётов (рендер)

В мессенджере **отчёт — это ответ, а не таблица**. Это совпадает с идеологией пилота «у вас нет
отчётов — просто спросите»: первичный продукт — `Answer.Text` (нарратив + ключевые числа, уже
посчитанные в Go), таблица вторична. Модель повторяет то, что доказал Петя: 3–5 строк текста +
xlsx-файл, а не таблица в чате.

`Render(ans) → (text, *document)` — **чистая функция, тестируется** (`render_test.go`):

1. **Всегда** — `Answer.Text`.
2. **`Envelope` пустой** (advice, динамика, один показатель) → только текст.
3. **Маленький** (≤ ~7–10 строк, ≤ 3 узких колонки) → текст + компактный моноширинный блок под
   мобильную ширину (~30–35 символов).
4. **Крупный/широкий или «выгрузи»** → текст со **сводкой** (Итого за период · Максимум-день ·
   Минимум-день, считается в Go из `Envelope.Summary`/`Rows`) **+ xlsx-документ** через готовый
   `export.XLSX` ([internal/export](../internal/export)).

Пороги (строк/колонок) — константы пакета `telegram`, крутятся без переписывания логики.

**Решение по v1:** сводка-текст + xlsx, **без** зависимости от веб-UI. Ссылка «открыть полный
отчёт» (deep-link на веб-UI) — добавится позже одной строкой, когда UI будет доступен по URL;
контракт не меняется. **Не делаем в v1:** рендер таблицы в PNG-картинку (лишняя зависимость,
нечитаемо скринридерами; xlsx + сводка достаточно).

## 5. План реализации (по этапам)

Новые файлы: `internal/transport/telegram/{bot,render,commands,render_test}.go`, `cmd/bot/main.go`,
`internal/bootstrap/bootstrap.go`. Затронутые: `internal/config/config.go`, `cmd/server/main.go`
(упрощён до HTTP, сборка ушла в `bootstrap`), перенос `exportFilename` в `internal/export` (+1
строка в `transport/http/server.go`), `go.mod`/`go.sum`. **Ядро — ноль правок.**

| Этап | Что | Готово, когда |
|---|---|---|
| 1. Конфиг + зависимость | `Telegram{Token, Allowlist []int64, DefaultTenant}` в `Config`/`Load`/`Summary`; `go get` | сборка чистая, пустой токен ничего не меняет |
| 2. Рендер | `Render` (§4) + перенос `exportFilename` в `export` | `render_test.go` зелёный |
| 3. Bot loop | `New`/`Run(ctx)`: polling, allowlist-guard, горутина+`typing` на апдейт, `Ask`→`Render`→отправка | отчёт даёт текст; «выгрузи» даёт xlsx; чужой chat_id (при allowlist) — отлуп |
| 4. Команды | `/start` `/help` (канонные), `/today` (синтез «как вчера»), `/tt КОД` (фильтр `sale_point` в сессии) | четыре команды отвечают |
| 5. Сборка | общая проводка `*app.App` → `internal/bootstrap`; `cmd/bot` запускает только бот, `cmd/server` — только HTTP; два сервиса в `docker-compose` | `go build ./...`+`go vet` чистые; каждый процесс рестартует независимо; пустой `TELEGRAM_TOKEN` → `cmd/bot` падает с явной ошибкой, API не задет |
| 6. Проверка + доки | команда запуска (env) + чек-лист ручных проверок; `go test ./...` зелёный | прогон запускает пользователь |

Объём этапов 1–5: **~2–3 дня**.

## 6. Границы (что НЕ закрывает бот)

- **`/tt` даёт UX и фильтр сейчас, но реальное переключение данных между точками** упирается в
  отложенную мультиточечность ([00-pilot-vision.md](00-pilot-vision.md), месяц 3). На пилоте одной
  точки `/tt` валидирует код, хранит выбор и подставляет фильтр; данные «переключатся» полноценно,
  когда заведём живой многоточечный источник. Код команды от этого не меняется.
- **Изоляция данных по тенанту не подключена.** Контекст тенанта (таймзона/валюта/каталог через
  `tenantctx.Lookup`) работает и влияет на ответ; но `dooglys.Query` не несёт `tenantID`, клиент
  данных — один глобальный инстанс с одними кредами. Per-tenant источник (карта `tenantID →
  dooglys.Client` + маршрутизация в `Fetch` + авто-рефреш) — работа месяца 3.
- **Голос (Whisper)** — отдельный этап ~2–3 дня (месяц 3): voice → STT → текст → `Ask`. Для первого
  демо не нужен.
- **Excel follow-up «добавь столбец»** — фича сессии/каталога, не транспорта.
