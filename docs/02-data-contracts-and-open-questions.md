# 02. Контракты данных и открытые вопросы

Этот документ — **чеклист того, что мы ждём от заказчика**, чтобы зафиксировать контракты,
собрать mock-данные и протестировать систему локально без реальных запросов.

Формат подачи для каждого пункта желателен такой: **пример запроса + пример ответа (как есть, реальный JSON)**.
Из этих примеров мы сразу делаем фикстуры для тестов (см. [03](03-testing-strategy.md)).

---

## 1. Что уже есть

### 1.1 Отчёт по чекам — `report/json/data?type=payment-total`
Параметры: `date_from`, `date_to` (формат `DD.MM.YYYY`), `type=payment-total`.
Ответ — массив объектов по дням. Пример полей одного дня:

```jsonc
{
  "date": "2026-06-18",
  "count": 3,              // кол-во чеков?
  "revenue": 307.74,       // выручка
  "profit": 267.05,        // прибыль (от чего считается? база себестоимости?)
  "average_revenue": 102.58,
  "average_sum": 102.58,
  "sum_all": 307.74,
  "revenue_card": 257.87, "revenue_cash": 49.87, "revenue_online": 0, "revenue_sbp": 0,
  "sum_card": 257.87, "sum_cash": 49.87, "sum_online": 0, "sum_sbp": 0,
  "return_count": 0, "return_sum": 0,        // возвраты
  "payback_count": 0, "payback_sum": 0       // ??? отличие от return
}
```

### 1.2 Отчёт по заказам — `report/json/dashboard-report?date_from=...&date_to=...`
```json
{ "orders_revenue": 615, "orders_count": 5 }
```

> Реальные примеры этих двух ответов сохранены в `docs/contracts/fixtures/` (TBD: добавить файлы).

## 2. Открытые вопросы — нужно получить

### 2.1 Авторизация Dooglys API ❗
- [ ] Базовый URL API (хост, версия).
- [ ] Как именно передаётся токен (заголовок? имя?).
- [ ] Механика refresh: endpoint обновления, срок жизни access/refresh, коды ошибок при протухании.
- [ ] Где мы храним токены тенантов (нам отдаёт пре-слой? или мы храним и рефрешим сами?).
- [ ] Формат ошибок API (HTTP-коды, тело ошибки).

### 2.2 Модель тенанта ❗
Что лежит в записи `tenant_id` (= id организации). Нам критично:
- [ ] **Таймзона** (Europe/Moscow и т.п.) — для резолва «сегодня/вчера».
- [ ] **Валюта** и точность (рубли? копейки? сколько знаков).
- [ ] **Точки/филиалы**: одна или несколько у тенанта? есть ли `point_id` и фильтр по точке?
- [ ] Связь нашего `tenant_id` ↔ организация/аккаунт Dooglys.

### 2.3 Семантика полей ❗
- [ ] Единицы: суммы в рублях или в минорных единицах? сколько знаков после запятой?
- [ ] `profit` — от какой базы (себестоимость есть в Dooglys)? может быть отрицательной (видно в данных) — почему?
- [ ] Разница `return_*` vs `payback_*`.
- [ ] `count` — чеки или позиции? `revenue` vs `sum_all` vs `sum_*` — точные определения.
- [ ] `revenue` это net (за вычетом возвратов) или gross?

### 2.4 Полный список отчётов / endpoints ❗ (ждём)
Для каждого нужно: URL, метод, параметры, поддерживаемые фильтры, пример ответа.
- [ ] Отчёт **по товарам** (упоминался как нужный для MVP) — контракт пока неизвестен.
- [ ] ABC-анализ (если есть готовый endpoint).
- [ ] Список доступных **фильтров** по каждому отчёту (категория, точка, сотрудник, канал оплаты).
- [ ] Есть ли «сырые чеки» (позиции внутри чека) — нужны для Class B «какие товары просели».

### 2.5 Технические свойства API
- [ ] Пагинация (есть ли, как устроена) и лимиты диапазона дат.
- [ ] Rate limits / квоты.
- [ ] Гранулярность данных (только по дням? есть по часам? по чекам?).

### 2.6 Контракт с пре-слоем авторизации
Что именно пре-слой передаёт нам в каждом запросе:
- [ ] `tenant_id`, `user_id`, `role` — формат и где (заголовки/тело).
- [ ] Как мы доверяем этому источнику (подпись/внутренняя сеть/JWT)?
- [ ] Идентификатор диалога/сессии для контекста и pending-state.

### 2.7 Будущее (фаза 4, фиксируем заранее)
- [ ] Какие операции записи планируются первыми (изменить цену, скрыть позицию…).
- [ ] Endpoints записи и их контракты.
- [ ] Модель ролей: кто имеет право на запись.

## 3. Что нам нужно, чтобы протестировать систему

Минимальный набор для полного локального прогона **без реального Dooglys**:

1. **Реальные примеры ответов** каждого endpoint (хотя бы по 1–2 на отчёт) → станут фикстурами.
2. **Описание полей** (раздел 2.3) → чтобы движок считал корректно.
3. **Пример записи тенанта** (таймзона, валюта, точки) → 2–3 mock-тенанта для проверки изоляции.
4. **Контракт пре-слоя** (раздел 2.6) → форма входного запроса к нашему API.

Имея это, мы поднимаем mock-сервер Dooglys на фикстурах и гоняем весь пайплайн локально
(детали — в [03-testing-strategy.md](03-testing-strategy.md)).
