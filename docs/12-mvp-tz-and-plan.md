# ТЗ и план разработки — MVP (Telegram, 3 тенанта)

> Согласовано с заказчиком (созвон 2026-07-01). Это итоговое ТЗ текущей фазы:
> только разработка приложения. Коммерция/оплата/инфраструктура заказчика — вне
> этого документа. Дедлайн демо: **пятница–суббота (2026-07-04/05)**.

## 1. Что делаем (scope)

1. **3 Telegram-бота** — по одному на тенанта.
2. **Безопасность бота** — whitelist пользователей (по `user_id`/телефону); строгая
   изоляция тенантов (нет утечки данных между тенантами).
3. **Логирование** запросов пользователей и ответов агента + оценка полезности.
   Для демо — без БД: уже есть `internal/querylog` (JSONL: вопрос→план→ответ) и
   `internal/feedback` (👍/👎, связан по `id`). БД откладываем до реального объёма
   (см. раздел 4, P1).
4. **Развёртывание** на отдельном сервере (Selectel); деплой через git, вызов
   моделей через внутренний роутер.
5. **Отчёты**, на которые бот отвечает (только эти 6):

| Отчёт (ТЗ) | Endpoint Report-API | Итог (total) |
|---|---|---|
| Выручка | `/report/payment` | `/report/payment/total` |
| Источники заказов | `/report/source-order` | `/report/source-order/total` |
| Товары и категории | `/report/products` + `/report/categories` | `/report/products/total` |
| Персонал (отчёт по кассирам) | `/report/personnel` | `/report/personnel/total` |
| Наличные в кассе | `/report/cash-on-hand` | — |
| Внесения и выплаты | `/report/cash-income-outcome` | — |

**Вне scope этой фазы (планировщик НЕ роутит):** advisor/консультант, запись (write),
а также отчёты, требующие отдельной обработки (см. инвентаризацию ниже).

### 1.1 Инвентаризация отчётов (`report.yml`)

Всего **36 endpoint'ов = ~20 самостоятельных отчётов** (остальное — под-вьюхи
`/total`, `/detail(s)`, `/summary`, `/groups`, `/data`). Разбивка по объёму работ:

| Уровень обвязки | Кол-во | Отчёты |
|---|---|---|
| **A. Полная интеграция (демо)** | 6 | payment, source-order, products, categories, personnel, cash-on-hand, cash-income-outcome |
| **B. Транспортно готовы** (path-map, но планировщик НЕ роутит до eval) | ~10 | type-order, expected-profit, paycheck, sales-on-map, order-payment-hour, special, special-products, order-processing-time, kitchen/cooks |
| **C. Отдельная обработка** (не сейчас) | 4 | rfm (+groups/summary), abc, customer-segmentation, customer-list |

**Почему так, а не «обвязать все 20 сразу».** Транспорт (добавить путь в
`reportPathMap` + sort-дефолт) — дёшев. Дорого и рискованно другое: на каждый отчёт
нужны каталог, `reportFilterColumn`, нарратор, **роутинг планировщика + eval-кейсы**.
Планировщик на плато 92.8% — завести ~14 новых маршрутов без корпуса под них = прямой
риск регресса к демо. Поэтому: транспорт делаем обобщённым (добавить GET-отчёт = строка
в map), полностью интегрируем 6 из ТЗ, ещё ~10 держим «транспортно готовыми» и
подключаем к планировщику по логам (P1).

**Уровень C несовместим с дженерик-GET-клиентом:** `rfm` требует 6 обязательных границ,
`abc` — обязательные `group_a/group_b`, `customer-segmentation`/`customer-list` — это
POST с телом запроса (а `customer-list` возвращает uuid'ы — внутренний хелпер, не отчёт).

## 2. Контракт авторизации (есть полный)

Два окружения, выбираются через ENV. **Для демо primary — внешний `api.dooglys.com`**
(не требует сетевого маршрута в кубы). В коде сейчас закрыт только внутренний
(x-context) — внешний `access-token`+`tenant-domain` нужно добавить.

**Внешний** (`https://api.dooglys.com`):
```
GET /api/v1/reports/report/payment?date_from=…&date_to=…&sort_by=date&sort_order=asc
access-token: <token>
tenant-domain: rukagreka
```

**Внутренний** (в кубах, `http://report.default.svc`):
```
GET /api/v1/report/payment?date_from=…&date_to=…&sort_by=date&sort_order=asc
x-context: {"tenant_id":"…","tenant_domain":"…"}
```

Отличия, критичные для конфига: внешний путь содержит префикс `/reports`, внутренний —
нет; внешний — заголовки `access-token`+`tenant-domain`, внутренний — `x-context`.
Спека всех методов — `docs/report.yml`.

## 3. Текущее состояние (база, 7.5/10)

Сильное: соблюдённая sandwich-архитектура (engine не знает про planner/narrator),
секреты не утекают в LLM (`plan.go` без tenant_id, `config.Summary()` режет пароли),
многослойная защита от инъекций (`refuse.go` 20+ regex, `validate.go` white-list),
зрелая eval-петля (615 кейсов планировщика, история в `bench/LEDGER.md`), 0 TODO/HACK.

Долг (концентрированный, предсказуемый):
- `internal/app/app.go` — god-object 833 строки (роутинг + confidence + session state).
- `internal/planner/refine.go` — 449 строк, 11 regex-рефайнеров, скрытая «вторая
  ступень планирования» (переопределяет метод модели). Хрупко.
- Боевая HTTP-обвязка отсутствует: нет rate-limit, нет `MaxBytesReader`, токен из
  `?key=` (утечёт в логи), `tenant_id` берётся из заголовков клиента.
- Нет CI.

Состояние Report-клиента (проверено в коде):
- `ReportAPIClient` ([reportclient.go](../internal/dooglys/reportclient.go)) — только
  `personnel`, только x-context.
- `payment`/`products` идут через `APIClient` (самосбор из заказов) — по решению
  пилота переключаемся на server-side агрегаты Report-API.

## 4. Работы (декомпозиция, привязана к приоритетам заказчика)

### P0 — блокеры демо (пятница/суббота)

1. **Слой авторизации.** Для демо — внешний режим: добавить в `ReportAPIClient`
   `access-token`+`tenant-domain` (база `api.dooglys.com/api/v1/reports`). Режим/база/
   префикс — через ENV (`DGS_REPORT_BASE`, `DGS_REPORT_AUTH=token|xcontext`); x-context
   оставить как второй режим (P1, когда поднимут маршрут в кубы). Проверить на боевом токене.
2. **Обобщённый транспорт + sort_by (БЛОКЕР).** В `report.yml` у каждого отчёта
   `sort_by: required: true` (свой enum на отчёт). Текущий `reportclient.go` шлёт
   только `date_from/date_to/page/per_page` → против боевого `api.dooglys.com` это
   **HTTP 400** (прошло в тестах лишь потому, что стаб `personnel` не требовал sort_by).
   Сделать клиент дженериком: дефолтный `sort_by`/`sort_order` на отчёт, единый
   GET-путь. Тогда добавить GET-отчёт = одна строка в map.
3. **Обвязать 6 отчётов ТЗ полностью** (уровень A): расширить `reportPathMap`/
   `reportLabel`/`reportFilterColumn` на `payment, source-order, products, categories,
   cash-on-hand, cash-income-outcome`; снять `payment/products` с самосбора `APIClient`;
   каждый продёрнуть на реальных данных, сверить с фикстурами. Плюс прописать ~10
   отчётов уровня B в path-map (транспортно готовы, планировщик НЕ роутит).
4. **3 тенанта + изоляция.** Конфиг трёх тенантов (домен + креды/контекст) через ENV.
   Whitelist пользователей на тенанта. Фича доступна только тенантам с реальной
   связкой (есть записи). Тест одновременно на трёх — нет кросс-утечки.
5. **Telegram как первичный канал**, но engine остаётся каналонезависимым: единый
   формат выхода (envelope), чтобы тот же движок встал в UI без переписывания.
6. **Логирование (без БД).** `querylog` + `feedback` уже пишут JSONL (Q→план→ответ
   + 👍/👎). Задача — убедиться, что они включены на всех 3 ботах и пишут tenant/user.
   БД не заводим (см. P1).
7. **Боевая HTTP-обвязка (безопасность):** rate-limit (защита LLM от спама —
   заказчик отметил риск «тенант лупит от души»), `MaxBytesReader` на теле, убрать
   `?key=`, реальный пре-слой auth для server-side `tenant_id` (не из тела клиента).
8. **Развернуть на Selectel** через git; подключить к внутреннему роутеру моделей.
9. **Абстракция вызова модели** — URL/модель не захардкожены, переключаемы через
   роутер по типу задачи (заложить, не реализовывать все ветки).

### P1 — сразу после демо (на собранных логах)

10. **Планировщик: мягкий гибкий слой.** Увести `refine.go` (regex-рефайнеры) в явный
    `plan.Classifier` — анализ задачи, а не постобработка. Описать в документации.
11. **Расширить роутинг планировщика** на ~10 «транспортно готовых» отчётов уровня B
    (из 1.1) — каждый с eval-кейсами, по логам, без скачка плато.
12. **Разнести `app.go`**: вынести session state-machine и confidence-роутинг.
13. **CI**: `go build/vet/test` + корпус-страж на PR (главный системный риск при
    текущей скорости коммитов).
14. **Write-path (предусмотреть).** Сейчас всё read-only; заложить точку расширения
    под запись, не реализуя.
15. **БД для логов — решение на объёме.** Когда JSONL покажет реальный объём/форму
    запросов и понадобится RAG/аналитика/дашборды — мигрировать в БД (вероятно
    Postgres). До этого момента БД не нужна. Решение принимаем по собранным логам.
13. **Тесты**: `envelope`, multi-turn диалоги, бенчмарки на боевом железе.

## 6. Куда двигаемся дальше (после MVP)

Сбор живых логов → расширение планировщика под реальные вопросы (итерации) →
tool-движок вместо regex-рефайнеров → подключение того же engine в UI → далее по
