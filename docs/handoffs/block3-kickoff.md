# Задача: Блок 3 (боевая обвязка / безопасность) MVP Telegram-бота Dooglys

Контекст — `docs/12-mvp-tz-and-plan.md` (раздел 4 P0 #9 + раздел 8) и память
`active-dev-plan`. Блок 3 — закрыть боевые дыры.

## Цель
Сделать сервис безопасным для боевого демо на 3 тенанта: защита от спама LLM,
строгий whitelist, боевая HTTP-обвязка, graceful-деградация на сбой API.

## Контекст канала (важно для приоритетов)
Демо-канал — Telegram (3 бота). Спам-риск заказчика («тенант лупит от души») идёт через
Telegram → rate-limit нужен на Telegram-стороне. HTTP-транспорт (`cmd/server`) —
dev/eval/HTTP-фикстуры; хардненинг делаем, но уточни, выставлен ли HTTP наружу в демо.

## Текущее состояние (проверь актуальные строки, код мог сдвинуться)
- `internal/transport/telegram/bot.go`: `allowed()` — ПУСТОЙ allowlist открывает бота
  ВСЕМ. Есть кап параллелизма (≤8 одновременных Ask), но нет per-chat rate-limit.
- `internal/transport/http/server.go` (~212 стр.): нет `MaxBytesReader` на теле; токен
  принимается из `?key=`; `tenant_id`/`session_id` берутся из заголовков/тела клиента.
- `internal/app/app.go`: ошибка `client.Fetch()` уходит наружу, а не в человеческий ответ.
  Реестр App уже строгий: неизвестный тенант → `errUnknownTenant` (Блок 2).
- `internal/config`: `Validate`/`ValidateTelegram` есть (Блок 2) — расширяем.
- Сборка/тесты — docker: `make test`, `make vet`.

## Задачи

### S1. Rate-limit Telegram (анти-спам)
- Per-chat лимит (N запросов / окно) поверх капа параллелизма. Превышение → мягкий ответ
  «слишком много запросов, подождите», не тихий дроп. Тест лимитера (N ok, N+1 отбит).

### S2. Строгий ValidateTelegram
- В проде требовать НЕПУСТОЙ allowlist на каждого тенанта (иначе fail-fast на старте).
  Сигнал режима (например `APP_ENV=prod|dev`); в dev — прежнее «пусто = открыт». Тест на оба.

### S3. HTTP-хардненинг (server.go)
- `http.MaxBytesReader(w, r.Body, 1<<20)` в handleAsk/handleFeedback.
- Убрать токен из `?key=` — только заголовок (`X-Auth-Token`/`Authorization: Bearer`).
- Rate-limit (per-IP/token) ЛИБО делегировать прокси и задокументировать в `docs/DEPLOYMENT.md`.

### S4. Server-side tenant_id (HTTP)
- `tenant_id` не доверять из тела/заголовка клиента без проверки: HTTP пропускает в App
  только валидированный tenantID. Неизвестный → отказ (в App enforced; HTTP не обходит).

### S5. Graceful Fetch-error
- Ошибка `client.Fetch()` → человеческий ответ («не смог получить данные, попробуйте
  позже»), реальная ошибка в логах. Не отдавать raw error. Тест на стаб-клиенте с ошибкой.

## Правила
- `make test`+`make vet` зелёные перед коммитом. Рига НЕ нужно.
- Коммиты EN без Co-Authored-By, раздельные на задачу.
- НЕ трогать промпт планировщика/refuse-regex (P1, потянет полный eval).
- Sandwich: секреты/tenantID не в LLM. Прогоны — через `tee`.
