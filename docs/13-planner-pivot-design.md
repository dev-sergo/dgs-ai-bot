# 13 — Разворот планировщика: опорный дизайн (актуально на 2026-07-01)

> Документ-якорь для итерации «разворот планировщика на оркестрацию инструментов».
> Фиксирует понятия, риски и стратегию, обсуждённые до начала кодирования, чтобы
> следующая сессия не пересобирала контекст с нуля. **Кода ещё нет** — это дизайн.
>
> Связано: [01-architecture.md](01-architecture.md), [11-mvp-and-roadmap.md](11-mvp-and-roadmap.md)
> (§6a spike), [glossary.md](glossary.md).

---

## 0. TL;DR

- «Разворот» — это на самом деле **два разных решения**, которые нельзя путать:
  **(A)** single-step tool-планировщик + адаптер; **(B)** agent-loop с синтезом.
  Почти вся польза — в (A), почти весь риск — в (B).
- Порядок: **пилот на текущей арке → (A) по данным лога → (B) только по доказанному спросу.**
- Механизм параллельной разработки — **флаг + аддитивный пакет в одном дереве**,
  НЕ отдельная папка `v2/` (форк гниёт с первого дня пилота).
- Ров безопасности = **узкий реестр инструментов** (что можно дёрнуть) +
  **детерминированное ядро** (верность чисел и выводов). Обе оси нужны вместе.

## 1. Где мы сейчас (текущий планировщик)

Пайплайн в [app.Ask](../internal/app/app.go):

```
planner.Plan (LLM, json_object) → parsePlan → Refine (9 гардов)
  → Validate (catalog white-list) → executeReport (switch по Method) → narrator
```

Свойства:
- Планировщик выдаёт **один** `AnalysisPlan` ([plan.go](../internal/plan/plan.go)) — спека отчёта.
- Модель «думает» **внутренними ключами каталога** (`sum_all`, `sum_cash`, `name`, `payment`).
- Ошибки модели чинят **9 детерминированных гардов** ([refine.go](../internal/planner/refine.go)):
  `RefineAdvice / ChannelShare / EmployeeRanking / ProductContribution / Forecast /
  PaymentEntityReport / PaymentChannelFilter / TopNOrder / DefaultMethod`.
- LLM-клиент умеет только `response_format: json_object` — нативного tool-calling API
  (`tools`/`tool_choice`/`role:tool`) в [client.go](../internal/llm/client.go) **нет**.

«Негибкость», о которой речь: каждый новый случай = новый гард или правка схемы.

## 2. Целевое: tool-планировщик — что меняется

| | Сейчас | После разворота |
|---|---|---|
| Что выдаёт модель | один `AnalysisPlan` | список шагов `{steps:[{tool,args}]}` |
| Язык модели | внутренние ключи каталога | бизнес-словарь (`revenue`, `payment_type:cash`) |
| Кто переводит в ключи движка | сам LLM (+ гарды латают) | **адаптер** (детерминированный код) |
| Заземление сущности | имя в фильтре, резолв позже | явный шаг `resolve_entity` до запроса |
| Добавить отчёт/синоним | каталог + часто новый гард + промпт | правка **описания инструмента**, без кода |
| Многошаг («почему упало») | невозможно | форма `steps[]` допускает (фаза B) |

**Пример.** Запрос «выручка наличкой по Выксе за июнь».

Сейчас модель обязана знать ключи и квирки (`наличка→sum_cash`; у `payment` нет фильтра
`payment_type` — это колонки; ошибётся → чинит [RefinePaymentChannelFilter](../internal/planner/refine.go)):

```json
{"report":"payment","method":"plain","metrics":["sum_cash"],"group_by":["date"],
 "period":{"kind":"explicit","from":"01.06.2026","to":"30.06.2026"},
 "filters":[{"field":"sale_point","values":["Выкса"]}]}
```

После разворота модель говорит бизнес-словарём, ключей движка не касается:

```json
{"steps":[
  {"tool":"resolve_entity","args":{"name":"Выкса","kind":"sale_point"}},
  {"tool":"query_metrics","args":{"metric":"revenue","filters":{"payment_type":"cash"},"period":"июнь"}}
]}
```

Адаптер переводит `revenue`+`payment_type:cash` → `report=payment, metrics=[sum_cash]`.
Модель **физически не может** ошибиться с «payment_type как фильтр» → гард становится
**не нужен по построению**, а не «переносится».

## 3. Ключевые понятия этого разворота

**agent-loop** — цикл, где LLM вызывает инструмент, **смотрит на результат, решает
следующий шаг**, повторяет, пока не соберёт ответ. Нужен для многошаговых вопросов
(«почему упало» → выручка → разбивка по каналам → по товарам → свод). Сегодня **нет**:
[executeReport](../internal/app/app.go) — один `switch`, один запрос. Это фаза (B).

**single-step (A)** — один вызов инструмента, синтеза нет. Модель отдаёт один результат,
своих выводов «почему» не делает. ~80% пользы разворота при низком риске.

**Реестр инструментов** — закрытый набор того, что планировщик вправе дёрнуть.
Тот же принцип white-list, что и сейчас, поднятый на уровень выше: было «поле вне схемы →
план отклонён», станет «инструмент/аргумент вне реестра → отклонён».

**Адаптер** — детерминированный код `tool-call → AnalysisPlan`. Берёт на себя работу,
которую раньше делали гарды: перевод бизнес-словаря в ключи движка. Основной объём работы (A).

**Три словаря — не путать:**
1. **White-list инструментов и полей** (что разрешено) — в коде [catalog](../internal/catalog/catalog.go),
   план переезда на `configs/catalog.yaml`. Жёсткий словарь для валидации.
2. **Синонимы для модели** (revenue = выручка/оборот/заработали/касса…) — в **тексте
   описаний инструментов** (system-промпт). Его правка дала в spike рост 91→97% без кода.
3. **Маппинг в ключи движка** (`revenue→sum_all`/`amount`) — в **коде адаптера**.

**spike** — маленький одноразовый эксперимент с таймбоксом, чтобы снять технический риск
ПЕРЕД вложением. У нас [scripts/spike_toolcall.py](../scripts/spike_toolcall.py) +
`gen_cases.py` + `spike_run.py` замерили на живой локальной модели, тянет ли она
tool-calling. Результат — **98.9%** на 1000 кейсов ([11-mvp §6a](11-mvp-and-roadmap.md)).

## 4. Реестр инструментов под ТЗ (6 отчётов)

| Инструмент | Что делает | Покрывает из ТЗ |
|---|---|---|
| `query_metrics` | метрика из отчёта (report + metric + group_by + filters + period) | выручка, товары/категории, персонал, источники заказов |
| `resolve_entity` | имя → объект каталога (точка/товар/категория/сотрудник) | «по Выксе», «Бизнес-ланч» |
| `forecast_revenue` | прогноз по темпу | (сверх ТЗ, движок есть) |
| `cash_report` | балансовые отчёты кассы | наличные в кассе, внесения/выплаты |
| `clarify` | переспросить при неоднозначности | защита от догадок |
| `none` | болтовня / приветствие / не по теме | безопасность |

Cash-отчёты по форме — не таймсерия (баланс / приход-расход), поэтому, вероятно, отдельный
инструмент, а не общий `query_metrics`. Транспорт под все 6 уже размечен в
[reportclient.go](../internal/dooglys/reportclient.go) (`reportPathMap`, «Уровень A — 6 отчётов ТЗ»),
но каталог/планировщик их пока НЕ роутят.

## 5. Два риска и их защиты

Реестр инструментов защищает по оси **«что можно дёрнуть»**. Он НЕ защищает по оси
**«как модель истолковала результат»**. Это два разных риска.

### Риск (I) — выбор: модель не так поняла задачу → не тот инструмент / отчёт
- Пример: «сколько заработали наличкой» → модель взяла `orders_count` вместо `revenue`.
- **Видимый** (ответ «не про то»), **ловится**: eval, валидация, гарды; spike мерил → 98.9%.
- Есть и сейчас. Живёт во всех фазах. **Лёгкий.**

Как отслеживать (часть уже в коде):
- До исполнения: `confidence<0.5` → clarify; полоса `[0.5,0.7)` → бот проговаривает
  интерпретацию (`describePlan`, [confirm.go](../internal/app/confirm.go)) и ждёт «да»;
  [Validate](../internal/plan/validate.go) отклоняет вне white-list. В развороте + явный `clarify`.
- После: [querylog](../internal/querylog/querylog.go) пишет `text→intent→plan→outcome`;
  кнопки 👍/👎 в Telegram; eval-корпус меряет % верного выбора.

### Риск (II) — синтез: инструменты и числа ВЕРНЫЕ, но вывод НЕВЕРНЫЙ
- Пример: «почему упала выручка» → модель корректно дёрнула все разбивки, **все числа
  верные**, но заключила «упала из-за пиццы», когда пицца выросла, а провал — от разового возврата.
- **Невидимый**: числа сходятся → ответ выглядит достоверным. Автоматически почти не ловится.
- Появляется **только когда есть шаг синтеза**, т.е. в agent-loop (B).

Где дыра на уровне кода: сегодня нарратор получает envelope с **плейсхолдерами**, числа
подставляются шаблоном — модель их менять не может. В loop шаг «вот 3 result-set'а, напиши
вывод» породил бы **свободный текст модели** — причинное утверждение ничем не заземлено.

**Защита:** шага «свободный синтез» не должно существовать. Причинность считает **код**:
у нас уже есть `Contribution` в [engine/classb.go](../internal/engine/classb.go) — математически
раскладывает изменение по измерению. Модель лишь **решает** запустить разбор (риск I),
атрибуцию **считает движок**, нарратор проговаривает готовые числа. Инвариант:
**любое число и любая атрибуция в ответе обязаны трассироваться к вычислению движка,
а не к тексту модели.** Loop только **оркестрирует** детерминированные инструменты,
сам над сырыми числами не «рассуждает».

### Риск (III, стратегический) — соблазн дать модели слишком много
«Универсальный `query`» формально — «один инструмент из списка», но фактически возвращает
модели свободу и убивает ров. №III — это «**не предай собственную защиту**, раздув инструменты».
Инструменты держим **узкими**.

## 6. Что остаётся железным (не трогаем)

«Бутерброд» сохраняется: меняется мягкий слой сверху, **твёрдое ядро в середине — нет.**
- `Validate` против white-list ([validate.go](../internal/plan/validate.go)).
- Вся арифметика в движке ([engine](../internal/engine/engine.go)) — детерминированно.
- Числа в ответ только из плейсхолдеров envelope; нарратор их не пишет.
- Tenant-скоуп и PII-strip.
- `AnalysisPlan` остаётся **внутренним контрактом** (адаптер строит его), поэтому
  15 не-тестовых файлов, завязанных на него, не трогаются.

Меняется **передок** (как модель формулирует намерение), а не движок и гарантии точности.

## 7. Стратегия параллельной разработки: флаг, а не форк

Цель: не сломать пилот, быть готовым быстро переключиться, не терять время.
**Механизм — feature-flag в одном дереве, НЕ папка `v2/`.**

Почему не `v2/` (форк):
- «Железные части» на пилоте НЕ заморожены (+4 отчёта, cash-формы, фиксы по фидбеку) →
  копия немедленно расходится с боевой веткой → ручная синхронизация = долг.
- Форк дублирует движок → две копии вместо одной.

| | Папка `v2/` (форк) | Флаг + доп. пакет (одно дерево) |
|---|---|---|
| Общие части | две копии, ручная синхронизация | одна копия, синхронно |
| Пилот в проде | отдельно | `PLANNER_MODE=legacy` (дефолт), не тронут |
| Переключиться на V2 | мердж форка (боль) | сменить env-переменную |
| Мердж-долг | копится | нет |

Как правильно:
- Новый код — **аддитивный пакет** `internal/planner/tool/` (новый планировщик + адаптер).
  Существующий планировщик не трогаем.
- Выбор — флаг `PLANNER_MODE=tool|legacy` в config/bootstrap.
- Общее (catalog, `AnalysisPlan`, validate, engine, narrator, session, querylog) —
  **переиспользуется, не копируется**.
- Разработка — ветка `feat/planner-v2`, частый ребейз на main, влить срез за срезом
  с флагом **по умолчанию off**. Main всегда релизится, пилот цел.

Это branch-by-abstraction / feature-flag — стандартный способ строить большое изменение
параллельно, не форкая прод.

## 8. Последовательность (фазы) и eval-гейты

**Фаза 0 — пилот на текущей арке.** Приоритет усилий. 6 отчётов + 3 бота + деплой.
Пилот — источник данных: включённый лог запросов покажет, ЧЕГО не хватает.

**Фаза A — single-step tool-планировщик за флагом** (по данным лога):
- Ч0 (базлайн): eval-host на legacy → число-якорь.
- Ч1: tool-планировщик + адаптер за флагом, legacy по умолчанию. Гейт: eval legacy = базлайн
  (старый путь не тронут) + spike-харнесс на tool-режиме.
- Ч2: флип default на tool-режим, гарды ещё включены. Гейт: полный eval-host vs базлайн.
- Ч3: снять **структурные** гарды первыми (моот по построению): `PaymentEntityReport`,
  `PaymentChannelFilter`, `DefaultMethod`. Семантические — позже, под корпус-страж.

Дисциплина: **прогон между серьёзными переделками** (риг гоняет пользователь);
регресс доказывать корпус-стражем, не абсолютным числом (eval-host шумит ~9 кейсов).

**Фаза B — agent-loop + синтез** — **только по доказанному спросу** (лог показал, что
владельцы реально задают многошаговые вопросы) и **только с детерминированным синтезом**.

## 9. «Будет узко» — развилка, которую решает лог

Негибкость проявится быстро, но «узко» бывает двух видов, и только один лечится разворотом:
- **(а) Не хватает отчётов/фич** («дайте склад», «сравните две точки») — это **ширина**.
  Разворот делает её дешевле, но каждую фичу всё равно добавляешь; на текущей арке тоже можно.
- **(б) Не понимает, как спрашивают** («почему упало», составные, follow-up, кривые формулировки)
  — это **гибкость понимания**, прямая вотчина разворота.

**Лог запросов решает, какого вида узость** → что приоритезировать в V2. Не строить V2
в предположении «узко = нужен разворот», пока лог не докажет, что узость в формулировках,
а не в отсутствии фич. Иначе риск построить не тот V2.

## 10. Открытые вопросы / решения к началу итерации

- Граница `query_metrics` vs отдельный `cash_report` для балансовых отчётов.
- Форма `steps[]` и парсер (fail-closed, как текущий [parsePlan](../internal/planner/llm.go)).
- Полнота словаря адаптера (метрики/разрезы/каналы/фильтры) — главный источник риска (II→адаптер).
- Нужен ли новый eval-корпус под tool-выбор (текущий заточен под `AnalysisPlan`).
- Нативный tool-API (`tools`/`role:tool`) в клиенте — НЕ требуется для (A): spike доказал
  подход через `json_object` + `steps[]`. Вопрос всплывает только на (B).
