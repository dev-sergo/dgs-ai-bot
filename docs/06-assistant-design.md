# 06 — Дизайн ассистента-консультанта

Статус: **дизайн** (2026-06-23). Код ещё не написан — документ фиксирует контракт и план,
чтобы синхронизировать видение до разработки. Метрика готовности — отдельный advice-eval (§6).

## 1. Зачем

Сейчас бот отвечает по схеме **«один вопрос → один отчёт → таблица + 1 фраза»**. Это закрывает
фактологию («выручка за неделю», «топ товаров»), но не закрывает главную продуктовую ценность —
**консультацию**: объяснить, в чём проблема, и подсказать, что делать.

Реальные запросы из прогона `pipeval-quality`, которые сейчас отбиваются в clarify/off_topic:

| Запрос | Сейчас | Цель (advice) |
|---|---|---|
| «на чём я теряю деньги?» | clarify | возвраты + скидки + убыточные позиции, с числами |
| «что убрать из меню?» | clarify | аутсайдеры по выручке/прибыли + вывод |
| «как поднять выручку?» | off_topic | где просел канал/товар/средний чек + 1–3 совета |
| «что мне улучшить?» | clarify | сводка слабых мест за период |

Консультант — **одна из ключевых фич проекта**, ради неё и затевался планировщик+нарратор.

## 2. Архитектурный разрыв

Текущий `app.Ask` ([internal/app/app.go](../internal/app/app.go)) — строго **один** `client.Fetch`
одним методом. Консультантский вопрос — это **«один вопрос → НЕСКОЛЬКО детерминированных выборок →
синтез + рекомендация»**.

Принцип сэндвича сохраняется без изменений: **числа считает Go, слова формулирует LLM**. Меняется
оркестрация: добавляется ветка, которая собирает «снимок бизнеса» (несколько `Fetch` + движок),
а advisor-LLM рассуждает поверх готовых чисел.

```
Planner(LLM, intent=advice) → BuildInsightBundle(Go: несколько Fetch + engine) → Advisor(LLM)
```

Консультант **не требует** системных доработок (авто-рефреш куки, мультитенантность) — он работает
поверх существующего data-слоя (fixture/http) и движка. Это осознанно вне скоупа (§7).

## 3. Insight-bundle — ядро фичи

Детерминированная функция: `(report-set, period, filters) → InsightBundle`. Собирает фиксированный
набор фактов о бизнесе за период из УЖЕ существующих выборок и движковых функций
([internal/engine](../internal/engine)). Все числа — источник истины, как в envelope.

Предлагаемый контракт (черновик, `internal/insight`):

```go
type Trend struct {
    Now, Prev, DeltaAbs, DeltaPct float64
    EmptyPrev bool // prev-период пуст → проценты не считаем (как empty_prev в contribution)
}

type Component struct { Label string; Now, Prev, Delta, Share float64 }
type NamedRow  struct { Name string; Quantity, Amount, Profit, Discount float64 }

type InsightBundle struct {
    Period, PrevPeriod envelope.Period
    Currency           string

    Revenue   Trend        // sum_all: текущий vs предыдущий
    AvgCheck  Trend        // взвешенный средний чек (derived, §5.1)
    Returns   Trend        // return_count / return_sum
    Discounts float64      // сумма скидок за период (products.discount_sum)

    ChannelMix   []Component // карта/наличные/онлайн/СБП: вклад в изменение выручки
    TopProducts  []NamedRow  // по amount desc, N=5
    BottomProducts []NamedRow // по amount asc (ненулевые продажи), N=5
}
```

Сборка переиспользует то, что уже есть:
- `Revenue`/`ChannelMix` → `engine.Contribution(payment, sum_all, …)` (раскладка по каналам уже готова);
- `TopProducts`/`BottomProducts` → `Fetch(products)` + сортировка (как в top_n);
- `Returns`/`Discounts` → суммы полей из `Fetch(payment)`/`Fetch(products)`;
- `AvgCheck` → derived-метрика (§5.1, взвешенное среднее, не наивная сумма).

**80% ценности фичи — здесь.** Как только есть детерминированный снимок, и нарратор, и консультант —
это просто разные промпты поверх одного объекта.

## 4. Маршрутизация: intent=advice

Планировщик получает новый intent (или метод) `advice`. Распознаёт «что улучшить / на чём теряю /
что убрать из меню / как поднять выручку / дай совет / где проблема / что оптимизировать».

**Маршрутизация — детерминированным узким guard'ом** (по образцу `RefineEmployeeRanking`/
`RefineProductContribution` в [internal/planner/refine.go](../internal/planner/refine.go)),
**не** фаззи-правилом в промпте. Урок проекта зафиксирован в журнале roadmap: фаззи-правило
в промпте растекается на соседей и роняло score 523→507. Узкий regex-guard набор фраз →
`intent="advice"`, период берётся как обычно (нет периода → clarify, как сейчас).

Безопасность не меняется: advice — это про ЭТО заведение за период; «дай совет по бизнесу вообще»
остаётся off_topic (нет привязки к данным).

## 5. Advisor-промпт (надстройка над нарратором)

Отдельный промпт ([internal/narrator](../internal/narrator) или новый `internal/advisor`): на входе —
`InsightBundle` (готовые числа), на выходе — объяснение + **1–3 конкретные рекомендации**.

Жёсткий grounding (тот же принцип, что у нарратора после фикса `metric_label`):
- опираться ТОЛЬКО на числа bundle, ничего не выдумывать и не пересчитывать;
- называть конкретные драйверы с цифрами («возвраты съели N ₽», «позиция X в минусе»);
- при пустом prev-периоде — не считать проценты, честно сказать о неполноте;
- только русский (см. баг 5.5 — qwen срывается в китайский; нужен пост-фильтр + fallback на `Compose`).

Ответ ложится в существующий `Answer.Text`; опционально — `Answer.Envelope` с 1–2 опорными
мини-таблицами (топ-дно товаров, каналы). Контракт `Answer` не меняется.

## 6. Eval для совета — самый недооценённый кусок

Текущие бенчмарки проверяют поля плана (`eval-host`) и подстроки ответа (`pipeval`). Свободный совет
так не отрегрессишь. **Но числа в bundle детерминированы** (данные из фикстур), поэтому совет
проверяем «must-mention»-ассертами против известного снимка:

```jsonc
{"query": "на чём я теряю деньги за прошлый месяц",
 "expect": {"intent": "advice",
            "contains_any": ["возврат", "скидк"],   // реальный драйвер потерь в фикстуре
            "mentions_number": true,                  // совет подкреплён цифрой
            "not_contains": ["средний чек составил"]}}// не подменяет тему
```

Это даёт регрессионную защиту без LLM-judge. Опционально позже — LLM-judge по рубрике
(связность/тон/уместность) для качественной оценки, но не как gate.

## 7. Явно вне скоупа консультанта

- **Авто-рефреш куки (п.2)** и **мультитенантность (п.3)** — ортогональны; консультант работает
  на fixture-данных для демо. Откладываем (приоритет заказчика — рабочий MVP-фич, не инфраструктура).
- **Композируемый query-AST (Этап 6)** — консультанту не нужен; bundle — это фиксированный набор
  выборок, а не произвольный запрос.

## 8. План и оценка

| Шаг | Что | Оценка |
|---|---|---|
| 5.1 | Derived-метрики (взвеш. средний чек, % возвратов, доля канала) | 1–2 дня |
| ядро | `internal/insight.BuildInsightBundle` + сборка из движка | 2–3 дня |
| route | intent=advice + детерминированный guard в refine | 1 день |
| advisor | advisor-промпт + grounding + русский-фильтр | 1–2 дня (итерации на живой LLM) |
| eval | advice-кейсы (must-mention против фикстур) | 2–3 дня |

**Полный MVP: ~1.5–2 недели.** **Тонкий демо-срез (1 тип вопроса, фикс. bundle): ~3–4 дня** —
уже выглядит как «он реально советует».

**Рекомендуемый первый срез:** «на чём я теряю деньги / что улучшить» → bundle{Revenue, Returns,
Discounts, BottomProducts} → advisor. Минимум выборок, максимум «вау» для заказчика.
